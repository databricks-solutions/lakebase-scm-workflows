# Design Specification: `lakebase-tdd-workflows`

**Status:** Draft (signed off in conversation; this is the canonical record)
**Repo:** `databricks-solutions/lakebase-app-dev-kit`
**Depends on:** `lakebase-scm-workflows` (paired branches), `lakebase-release-workflows` (tier model + deployment), `software-design-principles` (foundational engineering canon — see §13)
**Authors:** Kevin Hartman + Isaac

## 1. Purpose

Provide a substrate skill that adapts canonical Test-Driven Development to the paired git + Lakebase branch model. TDD here means *design through testing*: the act of writing a test forms the consumer-perspective usage pattern. The substrate makes experimental TDD economical (cheap branches, real databases per branch, parent-aware diffs) and gives the Product Owner a portable, tool-agnostic spec format.

The "workflows" suffix is load-bearing: this skill ships orchestration — handoff between roles, observability of workflow state, gated transitions between phases — not just primitives.

## 2. Non-goals

- **Not** a project management skill. JIRA / Linear / GitHub Issues are *adapters* that consume the spec format. They do not define it.
- **Not** a general engineering canon. SOLID, DRY, DTSTTCPW, clean code, layered architecture, and cross-cutting-concern ownership live in `software-design-principles` and are *imported*, not redefined here.
- **Not** a deployment skill. `/ship`, deploy targets, release-tier ceremony, and migrate-to-prod live in `lakebase-release-workflows` (see §17).
- **Not** a UI framework. The skill operates through CLI scripts + agent-readable SKILL.md + the existing SCM extension surfaces.
- **Not** a replacement for project-level slash commands. The project skills (`/design`, `/build`) survive as thin wrappers that compose substrate skills with project-specific concerns (JIRA, FEIP labels, Cursor ergonomics).

## 3. Context

Already landed:
- `lakebase-scm-workflows` — paired git + Lakebase branches, branch lifecycle, schema diff against parent, 3-tier release tooling.
- `lakebase-release-workflows` — branching + release methodology (production → staging → short-tier).

The gap: no codified TDD workflow built on top. Project skills (`/po`, `/driver-navigator-tdd`, `/dev-loop`) currently implement TDD ad hoc with project-specific conventions. We're extracting the universal substrate so any consumer (Claude Code, OpenAI Codex, Cursor, Databricks Genie Code, future projects) inherits the same discipline.

## 4. Lexicon

| Term | Definition |
|---|---|
| **Spike** | Throwaway exploration on a Lakebase branch. No test list, no rigor. Shortcuts allowed. Goal: learn. Code is never promoted as-is. Branch is deleted after notes are captured. |
| **Experiment** | A spike with rigor. Has a test list, runs the canonical TDD cycle, ends with working code + tests + learning. Branch is a candidate for promotion. |
| **Cycle** | One RED → GREEN → REFACTOR pass for a single test list item. |
| **Test list** | Beck's planning artifact. Ordered list of behavioral scenarios that define done for a feature. Lives at the feature level; per-AC views generated by transformation. |
| **AC (Acceptance Criterion)** | One observable behavior. Tagged `[API]` / `[E2E]` / `[Infra]`. Maps to one or more scenarios in the test list. |
| **Feature** | A user-facing capability. Has one or more stories, each with one or more ACs. Lives on one feature branch. |
| **Promote** | (N=1, or single-winner N≥2) Take an experiment as-is into the feature PR. Rename or fast-forward the experiment branch. |
| **Synthesize** | (N≥2 multi-winner) PO menu-picks capabilities across experiments; spec is renegotiated; a fresh TDD cycle on the synthesized spec produces the final feature code. |
| **N=1** | The iterative-refinement mode. One experiment branch, canonical TDD with PO feedback driving in-sprint refactors. **No promote/synthesize ceremony** — the branch IS the feature. |
| **N≥2** | The parallel-experiments mode. Multiple branches, deliberate race between strategies. Negotiated up front via the design-spec gate. |
| **Bad smell** | A pattern the orchestrator detects that suggests the workflow is sliding. Surfaces a proposed remediation to the HITL. |
| **Adapter** | Pluggable component that syncs the substrate's spec format to/from an external tracking tool (JIRA, Linear, GitHub Issues, plain MD). |

## 5. Roles

| Role | Responsibility | Played by (default) |
|---|---|---|
| **Spec Author** | Composes the initial draft spec — features, stories, ACs with `[API]` / `[E2E]` / `[Infra]` tags. Runs perspective analysis (Architecture, Security, UX, Data lenses). | Project skill `/design` |
| **Architect Reviewer** | Applies layering lens to the draft. Assigns each AC its layer + cross-cutting-concern ownership. Imports `software-design-principles` canon. | Substrate-shipped agent prompt |
| **Test Strategist** | Converts architecturally-annotated ACs into a Beck-style test list, ordered for design momentum. Emits per-AC views via transformation. | Substrate-shipped agent prompt (or Navigator PLAN mode) |
| **Orchestrator (Scrum-Master)** | Facilitates: runs the design-spec gate, cuts experiment branches, runs cycles in parallel up to budget, watches bad smells, presents outcomes to HITL. Never decides unilaterally. | Substrate-shipped agent |
| **Navigator** | Per Beck: writes failing tests, reviews. Forces interface decisions first. Never weakens an assertion. | Substrate-shipped agent contract; project skills may specialize |
| **Driver** | Writes minimal honest code to pass. Never deletes or weakens a test. | Substrate-shipped agent contract |
| **Product Owner (PO)** | Owns spec, ACs, test list ordering. Decides promote vs synthesize. Refines spec when bad smells signal drift. | HITL plays this; project skills can scaffold |
| **HITL** | Coordinates the loop. May play any/all roles solo. Owns every gate. The feedback loop they provide is essential. | The human |

Role separation in *artifacts* (test list, handoff files, selection log) is preserved regardless of who plays the role.

## 6. Spec storage format

The substrate ships a portable, tool-agnostic spec format. Every element has both human-readable markdown and machine-readable JSON. They stay in sync via `scripts/tdd/spec-sync.ts` which detects drift.

### 6.1 Directory layout

```
.tdd/
  spec.json                          ← top-level machine-readable index
  spec.md                            ← top-level human-readable overview
  workflow-state.json                ← current phase, current artifacts
  features/
    F1-partner-submits-assets/
      feature.md                     ← human-narrated description, design intent
      feature.json                   ← {id, name, nfrs[], success_metrics[], status, stories[], tdd_mode, experiment_count_default}
      architecture.md                ← (added by Architect Reviewer) layering decisions, pattern proposals
      test-list.md                   ← Beck's ordered test list (FEATURE-level, all stories + ACs)
      test-list.json                 ← machine-readable version
      stories/
        S1-submit-form/
          story.md                   ← user story prose
          story.json                 ← {id, asA, iWantTo, soThat, nfrs[], acs[]}
          acs/
            AC1.md                   ← human-narrated AC
            AC1.json                 ← {id, layer: API|E2E|Infra, given, when, then, scenarios[], nfrs[], architectural_notes, status}
            AC2.{md,json}
          scenarios/
            s-1.feature              ← Gherkin (frontmatter ties to AC)
            s-2.feature
          test-list-per-ac.json      ← generated transform: per-AC view of the master list
  experiments/
    F1-partner-submits-assets/
      exp-1-postgres-arrays/
        notes.md                     ← strategy summary, learning
        branch.txt                   ← Lakebase branch id
        outcomes.json                ← {tests_passed, schema_diff_summary, code_diff_lines, status}
        timeline.json                ← cycles + smell triggers + HITL interventions
      exp-2-json-blob/
        ...
  spikes/
    F1-explore-storage/
      notes.md
      branch.txt
  synthesis/
    F1-partner-submits-assets/
      synthesis-2026-05-26.md        ← menu-pick decision + integration rules
      synthesized-spec/              ← renegotiated spec, ready for fresh cycle
  cycles/
    F1/S1/AC1/
      cycle-001.json                 ← {test, gate_check, navigator_verdict, driver_changes, timestamp}
      cycle-002.json
  selection-log.md                   ← append-only HITL gate decisions + rationale
  smells.json                        ← detected smells + resolutions
  adapters/
    jira.json                        ← per-adapter config (optional)
```

### 6.2 Schemas

JSON Schema files in `scripts/tdd/schemas/`. Validated on every read by `scripts/tdd/spec-sync.ts`. Drift between MD narrative and JSON contract is warned, never auto-corrected.

### 6.3 Adapter interface

```ts
interface SpecAdapter {
  pushFeature(feature: Feature): Promise<{ externalId: string }>;
  pushStory(story: Story): Promise<{ externalId: string }>;
  pushAC(ac: AC): Promise<{ externalId: string }>;
  updateStatus(externalId: string, status: Status): Promise<void>;
  pull(externalId: string): Promise<Feature | Story | AC>;
}
```

Bundled adapters: `markdown.ts` (no-op — the spec IS the tracking), `jira.ts`, `github-issues.ts`. Project skills wire in the adapter they want.

## 7. Workflow phases

### 7.0 Discovery (Spec Author)

**Input:** business problem, success metrics, constraints, optional reference materials.
**Output:** draft `feature.md` + `feature.json`; `story.md` + `story.json` per story; `ac.md` + `ac.json` per AC. No test list yet. No architectural decisions yet.
**HITL gate:** **APPROVAL GATE 1 — Draft spec.** "Are these the right user-facing capabilities?"

### 7.1 Architectural review (Architect Reviewer)

**Input:** signed-off draft spec.
**Output:** same files, now with `nfrs[]`, `layer`, and `architectural_notes` populated. New `architecture.md` summarizing layering decisions and pattern proposals (drawn from `software-design-principles` canon).
**HITL gate:** **APPROVAL GATE 2 — Architectural lens.** "Are these the right layer assignments and patterns?"

### 7.2 Test-list construction (Test Strategist)

**Input:** architecturally-annotated spec.
**Output:** `test-list.{md,json}` per feature (Beck-canon, ordered for design momentum). Per-AC transformed views generated for agent consumption.
**HITL gate:** **APPROVAL GATE 3 — Test list.** "Does this ordering drive the design well?"

### 7.3 Design-spec gate (Orchestrator)

**Input:** signed-off spec + test list.
**Output:** experiment plan — `N` (1 or ≥2), strategies (if N≥2), resource budget (concurrent branches, wall-clock, agent-pair count). Recorded in `selection-log.md`.
**HITL gate:** **APPROVAL GATE 4 — Experiment plan.** "N=1, or race N=2? What's the budget?"

### 7.4 Implementation (Orchestrator + Navigator + Driver)

Per experiment branch:

```
Loop:
  Navigator: PLAN → RED (one test from list, against the experiment's branch DB)
  Driver:    GREEN (minimal honest code, against the experiment's branch DB)
  Navigator: REVIEW (design coherence + architectural concern check)
  Driver:    REFACTOR (if needed)
  Orchestrator: emit cycle artifact; watch smells; surface to HITL at boundary
  PO/HITL:   feedback at boundary; either continue, refine, or stop
Until: test list exhausted OR PO declares done OR bad smell triggers remediation
```

**N=1 outcome:** the branch IS the feature. PR for review. No promote/synthesize ceremony.
**N≥2 outcome:** orchestrator runs `compare-experiments.ts`; HITL/PO chooses **promote** (one wins as-is) or **synthesize** (menu-pick → renegotiated spec → fresh cycle on a new branch).

## 8. Handoff and observability

Three substrate guarantees:

### 8.1 Handoff (enforced by orchestrator)

Each phase has a defined predecessor + artifact contract. The orchestrator refuses to transition if the prior phase's artifacts are missing or invalid. Phase 1 won't start without a draft spec; Phase 2 won't start without architectural annotations; Phase 3 won't start without an ordered test list; Phase 4 won't start without an approved experiment plan in `selection-log.md`.

### 8.2 Observability (workflow-state surface)

```
.tdd/
  workflow-state.json          ← {phase, feature_id, story_id, ac_id, cycle_id, started_at}
  selection-log.md             ← append-only HITL decisions + rationale
  smells.json                  ← detected smells + resolutions
  cycles/<F>/<S>/<AC>/         ← per-cycle artifacts
  experiments/<F>/<exp>/timeline.json   ← per-experiment timeline
```

`scripts/tdd/status.ts` (+ `lakebase-tdd-status` CLI bin) reads all this and renders a single-screen summary. The SCM extension can read the same files for native sidebar surfaces.

### 8.3 Auditability

`selection-log.md` is append-only. Every HITL gate decision is recorded with timestamp + rationale + the artifacts it was based on. A team member coming in cold can read the log and understand the system's current state.

## 9. Bad-smell catalog

Orchestrator monitors continuously. On detection: pauses, proposes remediation, waits for HITL approval. Auto-detect, HITL-gated action.

| Smell | Signal | Proposed remediation |
|---|---|---|
| Test list drift | List grew by >25% since cycle start without HITL approval | PO refinement on spec |
| Cycle stall | N cycles in a row with no GREEN | Re-examine test ordering or spec ambiguity |
| API coherence drift | Same concept named differently across two consecutive PASS reviews | Rename refactor before next test |
| Fragility ratio | One behavior change failed >3 tests | Refactor + flag tests-mirror-implementation anti-pattern |
| Test cost spiral | Each subsequent test takes >2× the lines of the prior one | Reconsider boundary — outer-loop tests probably needed |
| Cross-experiment divergence (N≥2) | Two parallel experiments are solving different problems | Was an opinion gap hidden? Re-run design-spec gate |
| Dead-requirement signal | An AC has had no scenarios written in N cycles while others mature | Deprecate or clarify via PO refinement |
| Test deletion attempt | Driver or human attempts to remove or weaken an existing test | Hard block. Tests are immutable until the test list itself is renegotiated. |
| Boundary violation | Test references a private method or internal helper | Refactor to public boundary or move to inner-loop list |

Catalog lives in `scripts/tdd/smells.ts` as data; detector functions evaluate after every cycle.

## 10. Hard rules baked into the SKILL.md (agent contract)

1. Tests are immutable until the test list itself is renegotiated through the PO. Never delete or weaken a test to make it pass.
2. "Minimal code" means minimal *honest* code that satisfies the current test list, not just the current test. Use the test list as your horizon.
3. After every GREEN, ask: "would a fresh reader infer the right concept from this API now?" If no, request REFACTOR before the next test.
4. Test at the outermost public boundary that maps to the AC. Inner-loop unit tests are reserved for pure logic that can't be exercised through the outer boundary.
5. A correct refactor should not change the outer-boundary tests. A refactor that requires editing tests is suspect.
6. Never make a private method public to test it.
7. Test count is a lagging indicator. The leading indicator is "how cheap is the next test?" Rising cost = design problem.
8. Spike code is throwaway. Promote nothing from a spike branch into a TDD branch except notes.
9. N=1 mode is iterative refinement. There is no promote/synthesize ceremony — the branch IS the feature.

## 11. Integration points

- `lakebase-scm-workflows` — `createFeatureBranch` (experiments), `deleteBranch` (cleanup, HITL-gated), `getSchemaDiff` (comparison), `getConnection` (test DB per branch).
- `lakebase-release-workflows` — tier model (feature → staging → production), TTL conventions, "never delete production" rule. **Owns deployment + `/ship`** (see §17). TDD defers to release-workflows for anything past PR merge.
- `software-design-principles` — imported as canon. Navigator's PLAN must include an Architectural Concerns Mapping per the canon's table. Refactor heuristics come from the canon.

The TDD workflow composes these. It does not replicate any of them.

## 13. The `software-design-principles` sibling skill

A separate substrate skill (same repo, different folder) that codifies foundational engineering canon for TDD and other workflow skills to import. Required because TDD assumes practitioners already know what good engineering looks like.

Contents (Milestone 0):
- SOLID, DRY, DTSTTCPW
- Clean code (Uncle Bob principles, condensed and grounded)
- Layered architecture (infrastructure / service / HTTP / policy ownership)
- Cross-cutting concerns mapping (auth, authz, capability resolution, audit, rate limiting, schema, policy config — which layer owns each)
- NFR baseline checklist

Lives at `skills/software-design-principles/` in `lakebase-app-dev-kit`. Unprefixed (it is not Lakebase-specific). README's domain catalog grows a new bucket: **"Shared canon"**.

## 15. Open questions

1. **Spec versioning.** When the spec changes mid-flight, do we version individual files or rely on git history? Lean: git history is enough; synthesis log captures rationale.
2. **Multi-language test runner.** `run-cycle.ts` needs the project's test command. Source: `.tdd/config.json`, or detected from `package.json` / `pom.xml` / `pyproject.toml` like `/driver-navigator-tdd` does today? Lean: detection with override.
3. **Adapter mutation policy.** When a JIRA adapter is configured, do status transitions auto-push? Lean: auto-push on green-path transitions; HITL gate on destructive ones (close, archive).
4. **Synthesis spec inheritance.** Does the synthesized spec live in its own subtree or replace the original? Lean: subtree (`synthesis/<feature>/synthesized-spec/`) preserves history.

## 16. Out of scope (for this spec)

- Replacing `/design`, `/build`, project-skill orchestration.
- Building a dedicated UI. The SCM extension already provides branch comparison and Branch Diff Summary.
- Auto-deciding promote vs synthesize. Always HITL.
- Cross-feature dependency resolution.
- Deployment + release ceremony (see §17 — owned by `lakebase-release-workflows`).

## 17. Adjacent substrate work (deferred — `lakebase-release-workflows` extension)

This section captures decisions that affect the broader workflow surface but are scoped to a sibling substrate skill, not `lakebase-tdd-workflows`. Tracked here so the context is preserved for when the sibling work begins. **None of the TDD milestones (§14) depend on this section.**

### 17.1 `/ship` and deploy targets live in `lakebase-release-workflows`

Deployment is a release concern, not a TDD concern. `lakebase-release-workflows` already owns the tier model (production → staging → feature / test / uat / perf) and the release flow (cut-RC → regression → cut-backup → migrate-prod). Adding `/ship` + deploy targets to the same skill is a natural extension.

### 17.2 Composition: `/build` vs `/ship`

`/build` (TDD project wrapper):
- Runs cycles through to "feature is complete on the branch"
- Creates PR
- Merges PR
- Triggers the existing post-merge migration to production (CI-driven; already in place)

`/ship` (release-workflows project wrapper):
- Deploys to a named target (`staging`, `prod`, named Databricks App host, etc.)
- Reads `deploy-targets.yaml`
- Coordinates the full deploy sequence (build frontend → workspace upload → app create/update → secret sync → seed data, etc.)
- Owns the release-tier ceremony when applicable (cut-RC against current prod, regression test, cut-backup, migrate-prod)

Two paths between them:
- **Auto-merge path:** `/build` finishes → PR merges → post-merge CI runs the schema migration. That's a deployment in the sense of "the branch is now in production for the database," but the *app* still needs to be shipped if it's a Databricks App or similar host.
- **Explicit deploy path:** after `/build` finishes (or even mid-flight for staging review), invoke `/ship <target>` to deploy the app to that target.

`/build` does not auto-invoke `/ship`. Deployment is always explicit.

### 17.3 Future Milestone R0 (release-workflows)

Factor today's SCM extension deploy wizard logic (`deploy-targets.yaml`, workspace import, app create/update, secret sync, seed data) into `lakebase-release-workflows` so non-extension consumers (CLI users, agents) can invoke deployment with the same contract.

Scope:
- `scripts/release/deploy-targets.ts` — read / validate / write `deploy-targets.yaml`
- `scripts/release/ship.ts` — orchestrate the deploy sequence per target
- `scripts/release/release-tier.ts` — cut-RC, regression, cut-backup, migrate-prod helpers (factor from existing CI scripts)
- `skills/lakebase-release-workflows/SKILL.md` — agent contract for deployment
- `skills/lakebase-release-workflows/agents/shipper.md` — agent prompt for `/ship`
- BDD coverage

Not gated by TDD milestones. Can happen in parallel.

### 17.4 Other release verbs (potential sub-commands)

Standalone slash commands that may emerge alongside `/ship`:
- `/cut-rc` — cut a release candidate branch from production for regression testing
- `/cut-backup` — cut a backup branch before migrating production
- `/migrate-prod` — apply migration to production with rollback safety
- `/cut-tier` — cut a long-running tier (staging, uat, perf)

These map to existing release-tier methodology and would compose cleanly with `lakebase-scm-workflows` branch primitives.
